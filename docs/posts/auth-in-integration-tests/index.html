Mocking Authentication and Authorization in ASP.NET Core Integration Tests

12 Dec 2021

ASP.NET Core makes writing integration tests very easy and even fun. One aspect that might be a bit tough to figure out is authentication and authorization. We might want to run integration tests under different users and different roles.

To get started, let’s assume we have an endpoint like this:

1app.MapGet("hi", (HttpContext httpContext) =>
2{
3    var userId = httpContext.User?.Claims?.FirstOrDefault(c => c.Type == ClaimTypes.NameIdentifier)?.Value;
4
5    return $"Hello #{userId}";
6}).RequireAuthorization();

It’s a very simple endpoint. It gets the currently logged in user’s ID and says hello to them.

To make it possible to mock auth, we have to register a custom AuthenticationHandler.

Here is a simple implementation of a mock Authentication Handler:

 1public class TestAuthHandlerOptions : AuthenticationSchemeOptions
 2{
 3    public string DefaultUserId { get; set; } = null!;
 4}
 5
 6public class TestAuthHandler : AuthenticationHandler<TestAuthHandlerOptions>
 7{
 8    public const string UserId = "UserId";
 9
10    public const string AuthenticationScheme = "Test";
11    private readonly string _defaultUserId;
12
13    public TestAuthHandler(
14        IOptionsMonitor<TestAuthHandlerOptions> options,
15        ILoggerFactory logger,
16        UrlEncoder encoder,
17        ISystemClock clock) : base(options, logger, encoder, clock)
18    {
19        _defaultUserId = options.CurrentValue.DefaultUserId;
20    }
21
22    protected override Task<AuthenticateResult> HandleAuthenticateAsync()
23    {
24        var claims = new List<Claim> { new Claim(ClaimTypes.Name, "Test user") };
25
26        // Extract User ID from the request headers if it exists,
27        // otherwise use the default User ID from the options.
28        if (Context.Request.Headers.TryGetValue(UserId, out var userId))
29        {
30            claims.Add(new Claim(ClaimTypes.NameIdentifier, userId[0]));
31        }
32        else
33        {
34            claims.Add(new Claim(ClaimTypes.NameIdentifier, _defaultUserId));
35        }
36
37        // TODO: Add as many claims as you need here
38
39        var identity = new ClaimsIdentity(claims, AuthenticationScheme);
40        var principal = new ClaimsPrincipal(identity);
41        var ticket = new AuthenticationTicket(principal, AuthenticationScheme);
42
43        var result = AuthenticateResult.Success(ticket);
44
45        return Task.FromResult(result);
46    }
47}

The basic idea is this: by default authenticate every request with user id provided in the TestAuthHandlerOptions. If a test wants to send a request under on behalf of a different user, they can do so by sending the user ID in the UserId header of the HTTP request.

We also need to create a custom WebApplicationFactory that takes advantage of our mock Authentication Handler:

 1public class WebAppFactory : WebApplicationFactory<Program>
 2{
 3    public string DefaultUserId { get; set; } = "1";
 4
 5    protected override void ConfigureWebHost(IWebHostBuilder builder)
 6    {
 7        builder.ConfigureTestServices(services =>
 8        {
 9            services.Configure<TestAuthHandlerOptions>(options => options.DefaultUserId = DefaultUserId);
10
11            services.AddAuthentication(TestAuthHandler.AuthenticationScheme)
12                .AddScheme<TestAuthHandlerOptions, TestAuthHandler>(TestAuthHandler.AuthenticationScheme, options => { });
13        });
14    }
15}

We have defined a DefaultUserId property on the factory so that the individual test fixtures can specify their own default user ID.

And we can use the mock authentication in the test cases like this:

 1public class SimpleTest : IClassFixture<WebAppFactory>
 2{
 3    private HttpClient _httpClient;
 4
 5    public SimpleTest(WebAppFactory factory)
 6    {
 7        factory.DefaultUserId = "5";
 8
 9        _httpClient = factory.CreateClient();
10        _httpClient.BaseAddress = new Uri("https://localhost/");
11        // Use our mock Auth scheme 
12        _httpClient.DefaultRequestHeaders.Authorization = new AuthenticationHeaderValue("Test");
13    }
14
15    [Fact]
16    public async Task SayHiToNumber5()
17    {
18        _httpClient.DefaultRequestHeaders.Remove(TestAuthHandler.UserId);
19
20        var response = await _httpClient.GetStringAsync("hi");
21        Assert.Equal("Hello #5", response);
22    }
23
24    [Fact]
25    public async Task SayHiToNumber1()
26    {
27        _httpClient.DefaultRequestHeaders.Add(TestAuthHandler.UserId, "1");
28
29        var response = await _httpClient.GetStringAsync("hi");
30        Assert.Equal("Hello #1", response);
31    }
32
33    [Fact]
34    public async Task SayHiToNumber3()
35    {
36        _httpClient.DefaultRequestHeaders.Add(TestAuthHandler.UserId, "3");
37
38        var response = await _httpClient.GetStringAsync("hi");
39        Assert.Equal("Hello #3", response);
40    }
41}

And that’s it! With a few lines of code, you now have a flexible mock authentication scheme that you can use in your tests. You can also customize it to match your needs.
